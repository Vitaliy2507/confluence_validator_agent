# Confluence Validator Agent

Мультиагентная система для валидации аналитических страниц Confluence в
брокерском бизнесе. Проверяет структуру страницы по шаблону, извлекает
технические разделы (БД, REST API, Kafka), генерирует сводку изменений
через GigaChat и публикует отчёт комментарием на страницу.

Построено **только на стандартной библиотеке Python + `requests`**
(плюс `python-dotenv` для чтения `.env`) — без LangChain, BeautifulSoup,
Pydantic или иных фреймворков.

## Возможности

- Загрузка страницы Confluence и разбор storage-format HTML на секции
  (свой парсер на `html.parser`, без BeautifulSoup).
- Валидация обязательных/опциональных разделов по шаблону (с кэшированием
  правил шаблона на диске, TTL настраивается, форс-рефреш по флагу — см.
  ниже).
- Извлечение технических секций (ER-модель, интеграции, Kafka) и
  формирование сводки изменений через GigaChat.
- **Авторизация в GigaChat по `client_id`/`client_secret` с автообновлением
  токена** — access_token запрашивается и обновляется в фоне сам, руками
  подкладывать ничего не нужно (см. раздел ниже).
- Если валидация не пройдена — GigaChat **не вызывается** (экономия
  токенов), в комментарий уходит только отчёт об ошибках.
- Устойчивость к сбоям: retry с экспоненциальной задержкой для всех
  внешних вызовов; если GigaChat недоступен — отчёт всё равно публикуется,
  но без AI-сводки.
- Все исключения логируются с полным stacktrace в файл и консоль.
- `--dump-template-rules` — печатает распарсенные правила шаблона
  простой таблицей (без валидации страницы и без публикации комментария),
  чтобы быстро проверить, что парсер шаблона реально извлёк.

## Структура проекта

```
main.py                     — точка входа (CLI)
config/                     — настройки, промпты, кэш правил шаблона
core/
  orchestrator.py           — связывает всё вместе, строит отчёт,
                               dump_template_rules() для отладки
  agents/
    validator.py            — проверка структуры по шаблону
    extractor.py            — извлечение технических секций
    summarizer.py           — вызов GigaChat и сборка сводки
  template/
    loader.py                — загрузка/кэширование правил шаблона
    parser.py                 — разбор страницы-шаблона в правила
clients/
  confluence_client.py      — REST-клиент Confluence
  gigachat_client.py        — REST-клиент GigaChat (chat/completions)
  gigachat_auth.py          — OAuth2 client-credentials, автообновление токена
parsers/
  html_parser.py            — разбор HTML на секции (без BeautifulSoup)
  text_cleaner.py            — извлечение чистого текста
models/                     — датаклассы: Page, Section, TemplateRule,
                               ValidationResult, Summary
utils/                      — logger, retry (backoff), JSON-кэш
exceptions/                 — иерархия исключений API/валидации
tests/                      — unit-тесты + HTML-фикстура
```

## Установка

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# заполните .env реальными значениями CONFLUENCE_*, GIGACHAT_*, TEMPLATE_PAGE_ID
```

## Запуск

```bash
python main.py <confluence_page_id>
```

Отчёт публикуется комментарием на странице `<confluence_page_id>` и
дублируется в stdout / `validator.log`.

### Отладка: что реально распарсилось из шаблона

Вместо того чтобы скриншотить страницу-шаблон и разбирать её глазами,
можно напрямую посмотреть, что извлёк парсер:

```bash
python main.py --dump-template-rules
# или сразу со свежим фетчем страницы-шаблона:
python main.py --dump-template-rules --refresh-template
```

Команда не требует `page_id`, ничего не валидирует и не постит комментарий
— только печатает таблицу: номер, обязательный/нет, уровень заголовка,
ключевые слова, имя раздела и родитель (если есть), плюс итоговые счётчики.

## Как определяется обязательность разделов и когда шаблон перечитывается

Правила (`TemplateRule`, обязательные/опциональные разделы) хранятся в
кэше `config/template_rules.json` и переиспользуются между запусками, а
не парсятся заново на каждый вызов:

1. **Первый запуск** (или когда кэш отсутствует/повреждён) — правила
   загружаются со страницы-шаблона `TEMPLATE_PAGE_ID` и парсятся.
2. **Последующие запуски** — используется кэш, пока он не старше
   `TEMPLATE_CACHE_TTL` секунд (по умолчанию 86400 = 24 часа).
3. **После истечения TTL** — при следующем запуске правила
   перечитываются со страницы-шаблона автоматически, кэш обновляется.
4. **Принудительно прямо сейчас** (например, сразу после правки страницы-
   шаблона, не дожидаясь TTL) — запустите с флагом:

   ```bash
   python main.py <confluence_page_id> --refresh-template
   ```

   Либо просто удалите файл `config/template_rules.json` — при следующем
   запуске (даже без флага) кэша не будет, и правила перечитаются
   заново.

**Какие заголовки вообще становятся пунктами чек-листа.** Страница-шаблон
обычно содержит не только реальную структуру (`1.`, `2.`, `4.1`, `6.5` и
т.д.), но и иллюстративные примеры внутри некоторых разделов (например,
6.5 показывает, как заполнить *одно конкретное* функциональное требование,
со своими подзаголовками "Запрос"/"Ответ"/"Статусная модель", иногда даже
с собственной нумерацией "1.", "2." заново). Такие примеры не должны
становиться отдельными пунктами чек-листа. Используются два независимых
фильтра:
- заголовок должен быть пронумерован как `N` или `N.M` (не глубже,
  не без номера вообще);
- заголовок должен быть реальным заголовком 1-го или 2-го уровня (h1/h2)
  — примеры внутри раздела почти всегда оформлены заголовком глубже.

Оба фильтра нужны одновременно: примеры иногда проходят один фильтр (если
случайно совпадает номер), но крайне редко проходят оба сразу.

**Как определяется, обязателен раздел или нет.** Раздел считается
**обязательным только если на странице-шаблоне рядом с заголовком явно
написано** об этом — распознаются оба варианта: полная фраза
`Требование к ведению - обязательно`, и короткая пометка/бейдж прямо
после заголовка, в наречной (`обязательно`) или прилагательной
(`обязательный`/`обязательная`/...) форме. Всё остальное — явное
`опционально`/`опциональный`, отсутствие маркера, незнакомая
формулировка — трактуется как **опциональное**. Это осознанный выбор в
пользу безопасного дефолта: ложное "опционально" даёт максимум
предупреждение в отчёте, а ложное "обязательно" валит валидацию по
причине, которую автор страницы не видит и не может исправить.

В репозитории уже лежит предзаполненный кэш `config/template_rules.json`,
соответствующий структуре шаблона из спецификации, — так что агент
работает "из коробки" даже без доступа к реальной странице-шаблону при
первом запуске.

## Авторизация в GigaChat (автообновление токена)

Клиент больше не полагается на статический `GIGACHAT_TOKEN` — авторизация
идёт по OAuth2 client-credentials:

1. В `.env` указываются `GIGACHAT_CLIENT_ID` и `GIGACHAT_CLIENT_SECRET`.
2. `clients/gigachat_auth.py::GigaChatTokenManager` при первом запросе к
   GigaChat сам обращается к `GIGACHAT_AUTH_URL` с заголовком
   `Authorization: Basic base64(client_id:client_secret)` и получает
   `access_token` + `expires_at`.
3. Токен кэшируется в памяти процесса и переиспользуется для всех
   последующих вызовов, пока не приблизится к истечению (обновление
   заранее, за 60 секунд до `expires_at` — с запасом на сетевые задержки).
4. Если GigaChat всё же ответит `401` (токен отозван / протух раньше
   расчётного времени), клиент один раз принудительно обновляет токен и
   повторяет запрос — уже без вмешательства человека.
5. Ошибки эндпоинта авторизации (неверные `client_id`/`client_secret`,
   сетевые сбои) идут через тот же retry с экспоненциальной задержкой, что
   и остальные внешние вызовы, и поднимаются как `GigaChatAuthError`.

Никаких ручных действий с токеном не требуется — только `client_id` и
`client_secret` в `.env`, а дальше агент сам управляет своей сессией.

### Переменные окружения GigaChat

| Переменная              | Назначение                                              |
|--------------------------|----------------------------------------------------------|
| `GIGACHAT_URL`           | Базовый URL API комплишенов                              |
| `GIGACHAT_AUTH_URL`      | URL OAuth-эндпоинта для обмена credentials на токен       |
| `GIGACHAT_CLIENT_ID`     | Client ID из личного кабинета GigaChat                   |
| `GIGACHAT_CLIENT_SECRET` | Client Secret из личного кабинета GigaChat                |
| `GIGACHAT_SCOPE`         | `GIGACHAT_API_PERS` (физлицо) или `GIGACHAT_API_CORP`     |
| `GIGACHAT_MODEL`         | Имя модели (по умолчанию `GigaChat`)                      |
| `GIGACHAT_VERIFY_SSL`    | `true`/`false` — проверка TLS-сертификата (см. ниже)       |

> На хостах Sber GigaChat используется сертификат от Минцифры РФ. Если в
> вашем окружении нет соответствующего корневого сертификата в системном
> хранилище, запросы будут падать по TLS-верификации. Либо установите
> корневой сертификат Минцифры, либо (осознанно снижая безопасность)
> выставьте `GIGACHAT_VERIFY_SSL=false`.

## Тесты

```bash
python -m unittest discover -s tests -v
```

Тесты покрывают сценарии из спецификации: успешную валидацию, отсутствие
обязательного раздела, отсутствие опционального раздела, извлечение
технических секций (включая случай "секции нет / пусто"), политику
"по умолчанию опционально" в парсере шаблона (включая фильтрацию примеров
по номеру и по уровню заголовка, короткие и полные маркеры обязательности
в обеих грамматических формах), кэширование/TTL/форс-рефреш
`TemplateLoader`, `--dump-template-rules`, а также получение/кэширование/
автообновление GigaChat-токена (в т.ч. форс-рефреш после 401 и retry при
недоступности эндпоинта авторизации).

## Ограничения реализации

- Конвертация Markdown → Confluence storage-format в `confluence_client.py`
  умышленно минимальна: поддерживает только конструкции, которые реально
  генерирует построитель отчёта (заголовки, списки, `**bold**`,
  горизонтальные линии, абзацы) — это не универсальный Markdown-рендерер.
- Разбор страницы-шаблона (`core/template/parser.py`) полагается на два
  эвристических фильтра (номер заголовка + уровень заголовка) для отсева
  иллюстративных примеров, и на распознавание маркеров обязательности в
  ограниченном наборе форм (полная фраза, короткая пометка, наречие и
  прилагательное). Если реальная страница использует принципиально другую
  разметку (например, обязательность задаётся не текстом рядом с
  заголовком, а цветом/иконкой без текста, или структура вложенности
  отличается от `N`/`N.M`) — понадобится donастройка регулярных выражений
  под конкретный кейс. `python main.py --dump-template-rules` — самый
  быстрый способ проверить, что реально распозналось.
