# Confluence Validator Agent

Мультиагентная система для валидации аналитических страниц Confluence в
брокерском бизнесе. Проверяет структуру страницы по шаблону, извлекает
технические разделы (БД, REST API, Kafka), генерирует сводку изменений
через GigaChat и публикует отчёт комментарием на страницу.

Построено **только на стандартной библиотеке Python + `requests`**
(плюс `python-dotenv` для чтения `.env`) — без LangChain, BeautifulSoup,
Pydantic или иных фреймворков.

## Возможности

- Загрузка страницы Confluence и разбор storage-format HTML на секции
  (свой парсер на `html.parser`, без BeautifulSoup).
- Валидация обязательных/опциональных разделов по шаблону (с кэшированием
  правил шаблона на диске, TTL настраивается).
- Извлечение технических секций (ER-модель, интеграции, Kafka) и
  формирование сводки изменений через GigaChat.
- **Авторизация в GigaChat по `client_id`/`client_secret` с автообновлением
  токена** — access_token запрашивается и обновляется в фоне сам, руками
  подкладывать ничего не нужно (см. раздел ниже).
- Если валидация не пройдена — GigaChat **не вызывается** (экономия
  токенов), в комментарий уходит только отчёт об ошибках.
- Устойчивость к сбоям: retry с экспоненциальной задержкой для всех
  внешних вызовов; если GigaChat недоступен — отчёт всё равно публикуется,
  но без AI-сводки.
- Все исключения логируются с полным stacktrace в файл и консоль.

## Структура проекта

```
main.py                     — точка входа (CLI)
config/                     — настройки, промпты, кэш правил шаблона
core/
  orchestrator.py           — связывает всё вместе, строит отчёт
  agents/
    validator.py            — проверка структуры по шаблону
    extractor.py            — извлечение технических секций
    summarizer.py           — вызов GigaChat и сборка сводки
  template/
    loader.py                — загрузка/кэширование правил шаблона
    parser.py                 — разбор страницы-шаблона в правила
clients/
  confluence_client.py      — REST-клиент Confluence
  gigachat_client.py        — REST-клиент GigaChat (chat/completions)
  gigachat_auth.py          — OAuth2 client-credentials, автообновление токена
parsers/
  html_parser.py            — разбор HTML на секции (без BeautifulSoup)
  text_cleaner.py            — извлечение чистого текста
models/                     — датаклассы: Page, Section, TemplateRule,
                               ValidationResult, Summary
utils/                      — logger, retry (backoff), JSON-кэш
exceptions/                 — иерархия исключений API/валидации
tests/                      — unit-тесты + HTML-фикстура
```

## Установка

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# заполните .env реальными значениями CONFLUENCE_*, GIGACHAT_*, TEMPLATE_PAGE_ID
```

## Запуск

```bash
python main.py <confluence_page_id>
```

Отчёт публикуется комментарием на странице `<confluence_page_id>` и
дублируется в stdout / `validator.log`.

Первый запуск попробует загрузить и разобрать шаблон со страницы
`TEMPLATE_PAGE_ID`; результат кэшируется в `config/template_rules.json` на
`TEMPLATE_CACHE_TTL` секунд (по умолчанию 24 часа). В репозитории уже
лежит предзаполненный кэш, соответствующий структуре шаблона из
спецификации, — так что агент работает "из коробки" даже без доступа к
реальной странице-шаблону при первом запуске.

## Авторизация в GigaChat (автообновление токена)

Клиент больше не полагается на статический `GIGACHAT_TOKEN` — авторизация
идёт по OAuth2 client-credentials:

1. В `.env` указываются `GIGACHAT_CLIENT_ID` и `GIGACHAT_CLIENT_SECRET`.
2. `clients/gigachat_auth.py::GigaChatTokenManager` при первом запросе к
   GigaChat сам обращается к `GIGACHAT_AUTH_URL` с заголовком
   `Authorization: Basic base64(client_id:client_secret)` и получает
   `access_token` + `expires_at`.
3. Токен кэшируется в памяти процесса и переиспользуется для всех
   последующих вызовов, пока не приблизится к истечению (обновление
   заранее, за 60 секунд до `expires_at` — с запасом на сетевые задержки).
4. Если GigaChat всё же ответит `401` (токен отозван / протух раньше
   расчётного времени), клиент один раз принудительно обновляет токен и
   повторяет запрос — уже без вмешательства человека.
5. Ошибки эндпоинта авторизации (неверные `client_id`/`client_secret`,
   сетевые сбои) идут через тот же retry с экспоненциальной задержкой, что
   и остальные внешние вызовы, и поднимаются как `GigaChatAuthError`.

Никаких ручных действий с токеном не требуется — только `client_id` и
`client_secret` в `.env`, а дальше агент сам управляет своей сессией.

### Переменные окружения GigaChat

| Переменная              | Назначение                                              |
|--------------------------|----------------------------------------------------------|
| `GIGACHAT_URL`           | Базовый URL API комплишенов                              |
| `GIGACHAT_AUTH_URL`      | URL OAuth-эндпоинта для обмена credentials на токен       |
| `GIGACHAT_CLIENT_ID`     | Client ID из личного кабинета GigaChat                   |
| `GIGACHAT_CLIENT_SECRET` | Client Secret из личного кабинета GigaChat                |
| `GIGACHAT_SCOPE`         | `GIGACHAT_API_PERS` (физлицо) или `GIGACHAT_API_CORP`     |
| `GIGACHAT_MODEL`         | Имя модели (по умолчанию `GigaChat`)                      |
| `GIGACHAT_VERIFY_SSL`    | `true`/`false` — проверка TLS-сертификата (см. ниже)       |

> На хостах Sber GigaChat используется сертификат от Минцифры РФ. Если в
> вашем окружении нет соответствующего корневого сертификата в системном
> хранилище, запросы будут падать по TLS-верификации. Либо установите
> корневой сертификат Минцифры, либо (осознанно снижая безопасность)
> выставьте `GIGACHAT_VERIFY_SSL=false`.

## Тесты

```bash
python -m unittest discover -s tests -v
```

Тесты покрывают сценарии из спецификации: успешную валидацию, отсутствие
обязательного раздела, отсутствие опционального раздела, извлечение
технических секций (включая случай "секции нет / пусто"), а также
получение/кэширование/автообновление GigaChat-токена (в т.ч. форс-рефреш
после 401 и retry при недоступности эндпоинта авторизации).

## Переменные окружения

См. `.env.example`. Ключевые (полный список GigaChat-переменных — в
разделе выше):

| Переменная           | Назначение                                   |
|-----------------------|-----------------------------------------------|
| `CONFLUENCE_URL`      | Базовый URL Confluence                        |
| `CONFLUENCE_TOKEN`    | Персональный токен/пароль                     |
| `CONFLUENCE_USER`     | Email пользователя для Basic auth             |
| `GIGACHAT_CLIENT_ID`  | Client ID для OAuth client-credentials        |
| `GIGACHAT_CLIENT_SECRET` | Client Secret для OAuth client-credentials |

| `LOG_LEVEL`           | Уровень логирования (по умолч. `INFO`)        |

## Ограничения реализации

- Конвертация Markdown → Confluence storage-format в `confluence_client.py`
  умышленно минимальна: поддерживает только конструкции, которые реально
  генерирует построитель отчёта (заголовки, списки, `**bold**`,
  горизонтальные линии, абзацы) — это не универсальный Markdown-рендерер.
- Разбор страницы-шаблона (`core/template/parser.py`) ищет явные маркеры
  "Требование к ведению - обязательно/опционально" рядом с заголовком;
  если маркер не найден, раздел в чек-лист не попадает (лучше не
  угадывать, чем ошибочно требовать несуществующий раздел).
